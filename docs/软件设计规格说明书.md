# C++ 高性能 Web 服务器软件设计规格说明书

## 1. 引言

### 1.1 编写目的

本文档用于说明 C++ 高性能 Web 服务器课程设计项目的软件需求、总体设计、模块设计、接口设计、数据设计、运行环境、测试方案和性能目标。本文档既作为项目开发依据，也可作为课程设计报告和答辩材料的基础。

### 1.2 项目背景

Web 服务器是互联网应用的基础组件，负责接收客户端 HTTP 请求，并返回对应的资源或动态处理结果。本项目要求在 Linux/Unix 环境下完成一个 Web 服务器程序，支持 GET、POST、HTML 页面响应、错误处理、CGI、数据库操作，并基于线程池和 epoll 等技术实现高并发。

### 1.3 项目名称

C++ 高性能 Web 服务器设计与实现。

### 1.4 目标用户

1. 课程设计评审教师。
2. 学习 Linux 网络编程和 C++服务器开发的学生。
3. 需要通过浏览器访问服务器页面的普通客户端用户。

## 2. 总体目标

本系统计划实现一个运行在 Linux 环境下的轻量级 Web 服务器。服务器能够监听指定端口，接收浏览器或 HTTP 客户端请求，根据请求方法和路径返回静态资源、动态 CGI 结果或数据库操作结果。在并发模型上，系统采用 epoll 进行 IO 多路复用，采用线程池处理业务任务，从而提高并发连接处理能力。

## 3. 需求分析

### 3.1 功能性需求

#### 3.1.1 GET 请求处理

服务器应支持 HTTP GET 请求。客户端请求静态资源时，服务器根据 URL 路径在 webroot 目录下查找对应文件，并返回文件内容。

示例：

```text
GET /index.html HTTP/1.1
```

服务器应返回 index.html 页面。

#### 3.1.2 POST 请求处理

服务器应支持 HTTP POST 请求。客户端通过表单提交数据时，服务器应读取请求体，解析表单参数，并返回处理结果页面。

示例：

```text
POST /cgi/echo HTTP/1.1
Content-Length: 23

name=Tom&message=Hello
```

#### 3.1.3 静态资源访问

服务器应支持以下类型资源：

1. HTML 页面。
2. CSS 文件。
3. JavaScript 文件。
4. 图片资源。
5. 普通文本文件。

服务器应根据文件后缀设置合适的 Content-Type。

#### 3.1.4 错误处理

当客户端请求资源不存在时，服务器应返回 404 页面。当请求格式错误或服务器内部错误时，服务器应返回对应错误响应。

至少支持：

1. 200 OK。
2. 400 Bad Request。
3. 403 Forbidden。
4. 404 Not Found。
5. 500 Internal Server Error。

#### 3.1.5 CGI 运行

服务器应支持简单 CGI 运行机制。当客户端访问指定 CGI 路径时，服务器调用外部 CGI 程序，将请求参数传递给 CGI 程序，并将 CGI 输出结果作为 HTTP 响应返回给客户端。

示例流程：

1. 用户提交 HTML 表单。
2. 服务器解析 POST 参数。
3. 服务器调用 cgi-bin/echo_cgi。
4. CGI 程序生成 HTML 输出。
5. 服务器将结果返回给浏览器。

#### 3.1.6 数据库操作

服务器应支持通过页面对数据库进行增删改查。数据库优先采用 SQLite。

建议提供 users 表操作：

1. 新增用户。
2. 查询用户列表。
3. 修改用户信息。
4. 删除用户。

#### 3.1.7 高并发处理

服务器应使用 epoll 监听多个 socket 事件，避免一个连接阻塞整个服务器。同时使用线程池处理请求解析、文件读取、数据库操作等业务任务，提高并发处理能力。

#### 3.1.8 日志记录

服务器应记录关键运行信息：

1. 服务器启动和关闭。
2. 客户端连接建立和关闭。
3. 请求方法和路径。
4. 响应状态码。
5. 错误信息。
6. 数据库操作结果。

### 3.2 非功能性需求

#### 3.2.1 性能需求

1. 支持多个客户端同时访问。
2. 在普通 Linux 虚拟机环境下能够完成基础压测。
3. 能输出 QPS、平均响应时间、失败请求数等指标。

#### 3.2.2 可维护性需求

1. 代码按模块划分。
2. 网络、HTTP、线程池、数据库、日志等模块相互独立。
3. 使用 CMake 管理构建。
4. 使用 README 说明编译和运行方式。

#### 3.2.3 可移植性需求

系统主要面向 Linux，依赖 epoll，因此不要求跨 Windows 运行。但代码应尽量减少对特定发行版的依赖。

#### 3.2.4 安全性需求

1. 限制静态资源访问范围，防止路径穿越。
2. 对数据库输入进行基本过滤或使用预编译语句。
3. 限制 CGI 可执行目录。
4. 对过长请求进行限制，避免异常占用内存。

## 4. 系统总体设计

### 4.1 系统架构

系统采用主从 Reactor 简化结构：

1. 主线程创建监听 socket。
2. 主线程使用 epoll_wait 监听连接事件和读写事件。
3. 新连接到达时，主线程 accept 并加入 epoll。
4. 请求可读时，主线程读取请求或将连接任务提交给线程池。
5. 工作线程解析 HTTP 请求并生成响应。
6. 响应发送完成后关闭连接或等待下一次请求。

简化架构如下：

```text
Browser / curl
      |
      v
TCP Socket
      |
      v
Epoll Event Loop
      |
      v
Thread Pool
      |
      +--> HTTP Parser
      +--> Static File Handler
      +--> CGI Handler
      +--> Database Handler
      +--> Logger
      |
      v
HTTP Response
```

### 4.2 请求处理流程

```text
客户端连接
    |
    v
accept 新连接
    |
    v
设置非阻塞
    |
    v
加入 epoll
    |
    v
epoll_wait 监听事件
    |
    v
读取 HTTP 请求
    |
    v
判断请求是否完整
    |
    v
提交线程池处理
    |
    v
解析请求方法和路径
    |
    +--> 静态文件
    +--> CGI
    +--> 数据库操作
    +--> 错误处理
    |
    v
生成 HTTP 响应
    |
    v
发送响应
    |
    v
关闭连接或等待下一请求
```

## 8. 关键算法和机制

### 8.1 epoll 事件循环

epoll 用于监听多个文件描述符事件。服务器将监听 socket 和客户端 socket 加入 epoll，当连接到达或数据可读时，epoll_wait 返回事件列表，服务器逐个处理。

优势：

1. 避免每个连接一个线程。
2. 可以同时管理大量连接。
3. 适合 Linux 高并发服务器。

### 8.2 线程池

线程池提前创建固定数量线程，避免每个请求都创建和销毁线程。主线程把请求处理任务放入任务队列，工作线程从队列中取任务执行。

优势：

1. 降低线程创建销毁成本。
2. 控制系统线程数量。
3. 提高并发请求处理能力。

### 8.3 HTTP 解析

HTTP 请求解析流程：

1. 按 CRLF 分割请求行和 Header。
2. 解析请求行得到 method、path、version。
3. 解析 Header 保存到 unordered_map。
4. 根据 Content-Length 判断 Body 是否完整。
5. 根据 method 和 path 分发处理逻辑。

### 8.4 CGI 调用

CGI 调用流程：

1. 判断请求路径是否以 /cgi/ 开头。
2. 查找 cgi-bin 下对应程序。
3. 创建子进程执行 CGI。
4. 通过管道传递输入和读取输出。
5. 父进程等待 CGI 完成。
6. 将 CGI 输出作为 HTTP 响应体返回。

## 9. 安全设计

### 9.1 路径安全

服务器必须防止路径穿越攻击，例如：

```text
GET /../../etc/passwd HTTP/1.1
```

处理方式：

1. 对 URL 路径进行规范化。
2. 禁止包含 .. 的路径。
3. 限制访问根目录为 webroot。

### 9.2 请求大小限制

服务器应限制请求头和请求体大小，防止恶意大请求占用内存。

建议：

1. 请求头最大 8 KB。
2. 请求体最大 1 MB。

### 9.3 数据库安全

1. 使用预编译 SQL。
2. 对输入长度做限制。
3. 对 HTML 输出做基本转义。
