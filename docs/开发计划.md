# C++ 高性能 Web 服务器开发计划

## 1. 项目目标

本项目计划在 Linux 环境下使用 C++ 开发一个轻量级高性能 Web 服务器。服务器需要支持 GET/POST 请求、HTML 页面响应、404 等错误处理、简单 CGI 运行、数据库增删改查，并基于线程池和 epoll 实现高并发处理能力。

项目定位不是复刻 Nginx，而是完成一个课程设计级别的可运行、可测试、可讲解的 C++ 网络服务器。最终成果应包含源代码、构建脚本、测试页面、数据库示例、压测结果、设计文档和答辩材料。

## 2. 功能范围

### 2.1 必须完成

1. Linux 环境下编译、运行和部署。
2. 使用 C++ 实现 TCP Socket 服务器。
3. 支持 HTTP GET 请求。
4. 支持 HTTP POST 请求。
5. 支持静态 HTML、CSS、JS、图片等资源访问。
6. 请求资源存在时返回对应 HTML 页面或文件内容。
7. 请求资源不存在时返回 404 页面。
8. 支持简单 CGI 调用，例如表单提交后执行程序并返回结果。
9. 支持通过页面对数据库进行增删改查。
10. 使用 epoll 实现 IO 多路复用。
11. 使用线程池处理连接或业务任务。
12. 提供日志输出和基本错误记录。
13. 提供压测结果和性能分析。

### 2.2 推荐完成

1. 支持配置文件，例如端口、线程数、静态资源目录、数据库路径。
2. 支持 400、403、404、500 等常见错误响应。
3. 支持连接超时管理。
4. 支持简单路由分发。
5. 支持数据库操作页面：查询、添加、修改、删除。
6. 支持表单提交、结果页面展示。
7. 支持 graceful shutdown，即优雅关闭服务器。

### 2.3 可选加分

1. 异步日志系统。
2. 定时器关闭超时连接。
3. 数据库连接池。
4. Keep-Alive 长连接。
5. 压测对比：阻塞版、epoll 版、线程池版。
6. 简单权限或登录功能。

## 3. 技术路线

| 模块 | 推荐技术 |
|---|---|
| 开发语言 | C++17 |
| 开发环境 | Ubuntu / Debian / CentOS |
| 构建工具 | CMake |
| 网络编程 | Linux Socket API |
| IO 模型 | epoll + 非阻塞 IO |
| 并发模型 | 线程池 |
| HTTP 协议 | 自定义 HTTP/1.1 简化解析 |
| CGI | fork + exec 或独立可执行程序调用 |
| 数据库 | SQLite 优先，MySQL 可选 |
| 日志 | 同步日志优先，后期可改异步日志 |
| 压测工具 | ab / wrk |
| 调试工具 | gdb、valgrind、strace、lsof、netstat/ss |

数据库建议优先选择 SQLite。SQLite 不需要额外部署数据库服务，课程设计阶段更稳，能降低环境配置和答辩演示风险。如果后期时间充足，再扩展 MySQL。

## 5. 四个月开发安排

### 第 1 阶段：基础准备与环境搭建

目标：建立 Linux/C++ 项目开发基础，搭好项目骨架。

任务：

1. 安装 Linux 虚拟机或准备云服务器。
2. 熟悉 Linux 常用命令、目录、权限、进程管理。
3. 安装 g++、CMake、gdb、SQLite、wrk 或 ab。
4. 学习 CMake 基本使用。
5. 建立项目目录结构。
6. 写出最小 C++ 程序并完成 CMake 编译。

交付物：

1. 能在 Linux 下编译运行的 C++ 项目骨架。
2. README 初稿。
3. 开发环境截图或记录。

### 第 2 阶段：阻塞版 HTTP 服务器

目标：先做出能通过浏览器访问的基础 Web 服务器。

任务：

1. 使用 socket、bind、listen、accept 建立 TCP 服务端。
2. 读取客户端请求。
3. 解析 HTTP 请求行：method、path、version。
4. 支持 GET 请求。
5. 根据 path 返回 webroot 下的静态文件。
6. 支持 index.html 默认首页。
7. 支持 404.html 错误页面。
8. 增加基础日志。

交付物：

1. 浏览器可以访问服务器首页。
2. 可以访问静态 HTML 文件。
3. 访问不存在资源时返回 404 页面。
4. curl 测试记录。

### 第 3 阶段：HTTP 解析与 POST 支持

目标：完善 HTTP 请求解析，支持表单提交。

任务：

1. 解析 HTTP Header。
2. 解析 Content-Length。
3. 支持 POST 请求体读取。
4. 支持 application/x-www-form-urlencoded 表单数据。
5. 编写 form.html 测试页面。
6. 将表单提交结果以 HTML 页面返回。
7. 完善 400、500 等错误响应。

交付物：

1. GET/POST 均可测试通过。
2. 表单提交后服务器能够返回结果页面。
3. HTTP 请求解析模块初步独立。

### 第 4 阶段：epoll 与非阻塞 IO

目标：将服务器从阻塞模型升级为 epoll 事件驱动模型。

任务：

1. 学习 epoll_create、epoll_ctl、epoll_wait。
2. 将监听 socket 设置为非阻塞。
3. 将客户端 socket 设置为非阻塞。
4. 使用 epoll 管理监听 socket 和客户端连接。
5. 实现连接读事件处理。
6. 实现连接写事件处理。
7. 处理 EAGAIN/EWOULDBLOCK。
8. 初步完成 Reactor 模型。

交付物：

1. epoll 版本服务器可以正常访问页面。
2. 多客户端同时访问不阻塞。
3. 能说明 epoll 相比阻塞模型的优势。

### 第 5 阶段：线程池与高并发处理

目标：引入线程池，提高并发处理能力。

任务：

1. 实现线程池类。
2. 使用任务队列保存待处理请求。
3. 使用互斥锁和条件变量协调工作线程。
4. 主线程负责 epoll 事件监听。
5. 工作线程负责 HTTP 请求处理和响应生成。
6. 处理线程安全问题。
7. 增加线程数量配置项。

交付物：

1. 线程池模块可独立说明。
2. 服务器支持多线程处理请求。
3. 能展示线程池任务流转图。

### 第 6 阶段：CGI 与数据库 CRUD

目标：完成题目要求中的 CGI 和数据库页面操作。

任务：

1. 设计 cgi-bin 目录。
2. 支持调用简单 CGI 程序。
3. 将表单参数传给 CGI。
4. CGI 返回 HTML 结果。
5. 使用 SQLite 建立 users 表或 messages 表。
6. 实现新增、查询、修改、删除页面。
7. 将数据库操作结果以 HTML 页面返回。

交付物：

1. 表单提交后能够执行 CGI 并返回结果。
2. 页面可以操作数据库。
3. 数据库表结构和测试数据完整。

### 第 7 阶段：测试、压测与优化

目标：验证服务器功能和并发性能。

任务：

1. 编写功能测试清单。
2. 使用 curl 测试 GET/POST/404/CGI/数据库操作。
3. 使用 ab 或 wrk 进行压测。
4. 记录并发数、请求数、QPS、平均延迟、失败请求数。
5. 分析性能瓶颈。
6. 修复明显崩溃和内存泄漏问题。
7. 使用 valgrind 检查内存问题。

交付物：

1. 功能测试结果。
2. 压测报告。
3. 性能分析说明。

### 第 8 阶段：文档、报告与答辩

目标：整理课程设计成果，准备答辩。

任务：

1. 完成软件设计规格说明书。
2. 完成课程设计报告。
3. 绘制系统架构图、请求处理流程图、线程池流程图。
4. 准备演示脚本。
5. 准备答辩 PPT。
6. 梳理可能被问到的问题。

交付物：

1. 源代码。
2. 可执行程序。
3. README。
4. 设计文档。
5. 测试和压测结果。
6. 答辩 PPT。
